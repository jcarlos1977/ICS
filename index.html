  • ICS – Dashboard Module (Summary)

    The ICS Dashboard provides a simple but effective real‑time overview of key operational metrics across the system. Although intentionally lightweight in its current version, it already delivers essential visibility into product activity, location usage, physical count notes, and movement trends. It is designed to grow and adapt as more modules and analytics are added to the platform.

    Key Metrics Displayed
    The dashboard presents several high‑level indicators that summarize the state of the inventory and system activity:

    1. Product Overview
    • Total number of products registered in the system
    • Active products currently in use
    • Inactive products not being used
    This helps users quickly understand the size and health of the product catalog.

    2. Location Overview
    • Total number of warehouse locations
    • Active locations currently being used
    • Inactive locations reserved for reference or historical purposes
    This provides a snapshot of warehouse capacity and utilization.

    3. Physical Count Notes
    The dashboard highlights whether any notes were added during recent physical counts.
    This allows supervisors to quickly identify items that require review, recounting, or follow‑up.

    4. Inventory Movement Summary
    The dashboard displays:
    • Total movements recorded today in the History module
    • Total movements recorded overall
    This gives immediate visibility into daily activity levels and long‑term operational volume.

    Interactive Charts
    At the bottom of the dashboard, users can visualize movement data through a dynamic chart component.
    The chart supports multiple display modes:
    • Bar Chart
    • Line Chart
    • Pie Chart
    • Doughnut Chart

    These charts are linked directly to the History module and can display:
    • Movements by Type (Inbound, Put Away, Ship, Adjustments, etc.)
    • Movements by User (Today)
    • Movements by User (All Time)
    • Inventory Counts by User
    This allows users to analyze operational patterns, identify workload distribution, and detect unusual activity.

    Current Purpose & Future Expansion
    Although the dashboard is intentionally simple at this stage, it serves as a foundation for future enhancements.
    The long‑term vision is to:
    • Provide module‑specific dashboards
    • Display KPIs for receiving, put‑away, shipping, and cycle counts
    • Integrate alerts, exceptions, and performance indicators
    • Offer customizable widgets for different departments

    For now, the dashboard acts as a clean, centralized snapshot of system activity, giving users a quick understanding of what is happening across ICS.

    The Inventory by Location module provides a detailed, real‑time view of stock distribution across every warehouse location. It is designed for operational accuracy, fast navigation, and high‑volume transaction processing, combining spreadsheet‑like usability with strict WMS‑grade validations.

    Key Capabilities

    Quick Filters & Advanced Filters
    Users can instantly narrow results or apply multi‑criteria filtering to analyze inventory by part, location, status, or any other attribute.

    Excel‑Style Interaction
    The table behaves like a modern spreadsheet:
    • Full‑row highlight when selected
    • Active‑cell highlight showing the exact clicked cell
    • Smooth hover effects for fast scanning
    This makes the interface extremely intuitive for warehouse operators.

    Export Tools
    Inventory can be exported to Excel or CSV for reporting, audits, or external analysis.

    Bulk Import Tools
    • Full inventory replacement during physical counts
    • Mass inventory adjustments without manual entry
    These tools dramatically reduce time and eliminate human error during large updates.

    Operational Functions (Real‑Time & Validated)
    This module is not just a table — it is an operational engine. It includes several real‑time inventory actions:

    Sub‑Transfer (Location‑to‑Location Movement)
    Move material from one location to another with automatic adjustments to both source and destination.
    Every movement is validated and recorded in the system history.

    Ship (Material Consumption / Picking)
    Remove inventory from a location in real time.
    The system logs:
    • What was taken
    • From which location
    • When
    • By which user
    All with strict validation to prevent incorrect part numbers or invalid locations.

    Receive (Add Inventory)
    Add material into a location with real‑time updates and full validation.
    Ensures that only valid part numbers and valid locations are used.

    Strict Validation Logic
    All operations in this module are protected by strong validation rules:
    • Prevents entering part numbers that do not exist in the system
    • Prevents using locations that are not registered
    • Ensures quantities are correct and consistent
    • Blocks invalid or incomplete transactions
    This guarantees data integrity and prevents operational mistakes.

    Real‑Time History Tracking
    Every action—ship, receive, sub‑transfer, bulk import, or physical count—automatically writes a detailed record into the system history, ensuring full traceability for audits and operational reviews.

    Counts are entered through a dedicated modal form that ensures accuracy and prevents invalid data entry.

    Validation Logic
    When the user enters a part number:
    • The system validates it against the Item List
    • If the part number is invalid, the system blocks the entry and alerts the user
    • Only valid part numbers are allowed

    Once validated, the modal automatically displays:
    • Unit of Measure
    • Description
    • Next Tag Number assigned to the user

    The user then enters:
    • Counted Quantity
    • Location

    After saving, the entry is written to the Physical Counts table.

    Tag Number Logic (Sequential & User‑Specific)
    Tag Numbers are essential for audit tracking. ICS assigns Tag Numbers using the following rules:
    • Each user receives their own sequence
    • Tag numbers include the user’s initials
    • Numbers increment sequentially (e.g., JS‑10000, JS‑10001, JS‑10002…)
    • No duplicates are allowed

    Tag numbers help auditors trace who counted what, and in what order.

    User Permissions & Visibility
    The module enforces strict visibility rules:
    • Users can only see their own counts — prevents bias or influence during the counting process
    • Administrators can see all counts — necessary for reconciliation, auditing, and oversight

    This separation ensures integrity during the counting phase.

    Discrepancy Tracking
    The system automatically compares:
    • Counted Quantity
    • System Quantity

    If there is a difference:
    • The discrepancy is calculated
    • The row is highlighted
    • The system quantity is displayed for reference

    This makes it easy to identify items that require investigation or recounting.

    Reconciliation Workflow (Controlled & Manual)
    The Physical Counts module does not automatically update the Inventory By Location table — intentionally.

    Before updating inventory, the reconciliation team must:
    • Review all discrepancies
    • Compare counts with system values
    • Discuss variances with finance
    • Approve or reject adjustments
    • Perform recounts if necessary

    Only after approval is the final inventory uploaded to replace the existing Inventory By Location data.

    This ensures financial accuracy and prevents accidental inventory corruption.

    • Quick Filters
    • Advanced Filters
    • Export to Excel / CSV

  • ICS – Módulo de Conteos Físicos (Resumen)

    El módulo de Conteos Físicos está diseñado para soportar conteos cíclicos precisos, auditorías completas de inventario y procesos de conciliación. Proporciona un entorno controlado, validado y completamente trazable donde los usuarios pueden registrar cantidades contadas, compararlas con los valores del sistema y preparar los datos para la aprobación final por parte de los equipos de inventario y finanzas.

    Campos Capturados en la Tabla de Conteos Físicos
    Cada entrada de conteo físico incluye los siguientes campos:
    • Número de Tag
    • Número de Parte
    • Descripción
    • Cantidad Contada
    • Fecha del Conteo
    • Hora del Conteo
    • Ubicación Contada
    • Contado Por
    • Discrepancia (diferencia entre contado vs. cantidad del sistema)
    • Cantidad del Sistema
    • Notas (editable para comentarios como “duplicado”, “ignorado”, “dañado”, etc.)

    Estos campos proporcionan una pista de auditoría completa para cada artículo contado.

    Flujo de Trabajo Basado en Modal
    Los conteos se ingresan mediante un formulario modal dedicado que garantiza precisión y evita entradas inválidas.

    Lógica de Validación
    Cuando el usuario ingresa un número de parte:
    • El sistema lo valida contra la Lista de Artículos
    • Si el número de parte es inválido, el sistema bloquea la entrada y alerta al usuario
    • Solo se permiten números de parte válidos

    Una vez validado, el modal muestra automáticamente:
    • Unidad de Medida
    • Descripción
    • Siguiente Número de Tag asignado al usuario

    El usuario luego ingresa:
    • Cantidad Contada
    • Ubicación

    Después de guardar, la entrada se escribe en la tabla de Conteos Físicos.

    Lógica del Número de Tag (Secuencial y Específica por Usuario)
    Los números de Tag son esenciales para auditoría. ICS los asigna bajo estas reglas:
    • Cada usuario recibe su propia secuencia
    • Los números incluyen las iniciales del usuario
    • Los números incrementan secuencialmente (ej. JS‑10000, JS‑10001…)
    • No se permiten duplicados

    Los números de Tag ayudan a rastrear quién contó qué y en qué orden.

    Permisos y Visibilidad del Usuario
    El módulo aplica reglas estrictas de visibilidad:
    • Los usuarios solo pueden ver sus propios conteos — evita sesgos o influencia
    • Los administradores pueden ver todos los conteos — necesario para conciliación y auditoría

    Esta separación garantiza integridad durante el proceso de conteo.

    Seguimiento de Discrepancias
    El sistema compara automáticamente:
    • Cantidad Contada
    • Cantidad del Sistema

    Si hay diferencia:
    • La discrepancia se calcula
    • La fila se resalta
    • La cantidad del sistema se muestra como referencia

    Esto facilita identificar artículos que requieren investigación o reconteo.

    Flujo de Conciliación (Controlado y Manual)
    El módulo NO actualiza automáticamente la tabla de Inventario por Ubicación — intencionalmente.

    Antes de actualizar inventario, el equipo de conciliación debe:
    • Revisar discrepancias
    • Comparar conteos con valores del sistema
    • Discutir variaciones con finanzas
    • Aprobar o rechazar ajustes
    • Realizar reconteos si es necesario

    Solo después de la aprobación se carga el inventario final para reemplazar los datos existentes.

    Esto garantiza precisión financiera y evita corrupción accidental del inventario.

    Herramientas de Exportación
    El módulo incluye:
    • Filtros Rápidos
    • Filtros Avanzados
    • Exportar a Excel / CSV

    Estas herramientas permiten exportar resultados para reuniones de conciliación, auditorías o revisiones externas.

    Funciones de Interacción del Usuario
    Como todas las tablas ICS, incluye:
    • Resaltado de filas
    • Resaltado de celda activa
    • Efectos hover
    • Interacción tipo hoja de cálculo

    • Validación estricta de números de parte

  • ICS – 实物盘点模块（摘要）

    实物盘点模块旨在支持精确的循环盘点、完整的库存审计和对账流程。它提供一个受控、验证且完全可追溯的环境，让用户记录盘点数量、与系统数量进行比较，并为库存与财务团队的最终批准做好准备。

    盘点表中记录的数据字段
    每条盘点记录包含以下字段：
    • 标签编号（Tag Number）
    • 物料编号
    • 描述
    • 盘点数量
    • 盘点日期
    • 盘点时间
    • 盘点库位
    • 盘点人
    • 差异（盘点数量 vs 系统数量）
    • 系统数量
    • 备注（可编辑，如“重复”、“忽略”、“损坏”等）

    这些字段为每个盘点项目提供完整的审计轨迹。

    基于 Modal 的盘点流程
    盘点通过专用的 modal 表单输入，确保准确性并防止无效数据录入。

    验证逻辑
    当用户输入物料编号时：
    • 系统会根据物料清单进行验证
    • 若无效，系统会阻止录入并提醒用户
    • 仅允许有效物料编号

    验证通过后，modal 会自动显示：
    • 计量单位
    • 描述
    • 分配给该用户的下一个标签编号

    用户随后输入：
    • 盘点数量
    • 库位

    保存后，记录写入盘点表。

    标签编号逻辑（顺序且用户专属）
    标签编号对审计至关重要。ICS 按以下规则分配：
    • 每个用户拥有自己的编号序列
    • 标签编号包含用户姓名缩写
    • 编号按顺序递增（如 JS‑10000, JS‑10001…）
    • 不允许重复

    标签编号帮助审计人员追踪谁在何时盘点了什么。

    用户权限与可见性
    模块执行严格的可见性规则：
    • 用户只能看到自己的盘点记录 — 防止偏见或相互影响
    • 管理员可查看所有记录 — 用于对账与审计

    这种分离确保盘点阶段的公正性。

    差异跟踪
    系统自动比较：
    • 盘点数量
    • 系统数量

    若存在差异：
    • 自动计算差异
    • 高亮显示该行
    • 显示系统数量作为参考

    便于快速识别需要调查或复盘的项目。

    对账流程（受控且手动）
    盘点模块不会自动更新“按库位库存”表 — 这是有意设计的。

    在更新库存前，对账团队必须：
    • 审查所有差异
    • 将盘点与系统数量进行比较
    • 与财务部门讨论差异
    • 批准或拒绝调整
    • 必要时进行复盘

    只有在批准后，最终库存才会上传并替换现有数据。

    这确保财务准确性并防止库存数据被意外破坏。

    导出工具
    模块包含：
    • 快速筛选
    • 高级筛选
    • 导出至 Excel / CSV

    这些工具便于团队导出盘点结果用于对账会议、审计或外部审查。

    用户交互功能
    与所有 ICS 表格一致，包含：
    • 行高亮
    • 活动单元格高亮
    • 悬停效果
    • 类电子表格交互方式

    为何此模块至关重要
    实物盘点模块确保：
    • 精确的循环盘点
    • 完整的盘点责任追踪
    • 严格的物料编号验证
    • 清晰的差异识别
    • 与财务部门的正确对账
    • 干净、可审计的盘点记录
    • 防止库存被意外修改

    它是 ICS 审计流程的核心，也是维持库存准确性与财务完整性的关键组件。
